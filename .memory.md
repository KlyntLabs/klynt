# Klynt Spec Review Memory

Lessons accumulated from spec reviews.

## 2026-06-18 — Frontend Foundation Design review

- **Security/privacy cannot be deferred implicitly.** When a spec scopes education-domain modules (grades, payments, users, admin), it must include at least a minimal Privacy & Data Governance section and a role model, even if backend enforcement is later. Future specs should treat FERPA/COPPA/GDPR and RBAC as first-class concerns, not afterthoughts.
- **Auth/session specs must resolve persistence before routing behavior.** Any design that says "tokens are in-memory" and "restore session on mount" in the same document is contradictory and will produce page-reload bugs. Pick a storage decision and propagate it to route guards and the test strategy.
- **Foundation specs need boundary rules, not just folder lists.** "Shared core + feature slices" only works if the spec defines allowed import directions, public surfaces, and where lazy loading/guards/fallbacks live. Without this, the core becomes a junk drawer.
- **Success criteria for foundation work need acceptance criteria and metrics.** Build/test gates (lint, typecheck, tests, build) are necessary but not sufficient; add measurable outcomes such as scaffold time, coverage thresholds, and behavior-based acceptance tests.
